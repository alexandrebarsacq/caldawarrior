Purpose: CalDAV clients like tasks.org show "midnight" instead of "all day" when a DATE-only value gains a spurious time component. Ambiguous DST times currently cause silent datetime loss. Both are correctness issues that affect real-world interoperability.

Output: Modified VTODO struct with date-only tracking, updated parser/serializer with conditional DATE-only output, DST fallback chain in TZID resolution, writeback propagation, and comprehensive unit tests proving all behaviors.

**Step 3: Fix DST handling in parse_datetime_with_params in src/ical.rs**

Replace line 454:
```rust
let dt = tz.from_local_datetime(&naive).single()?;
```

With the fallback chain:
```rust
let mapped = tz.from_local_datetime(&naive);
let dt = mapped.single()
    .or_else(|| mapped.latest())   // fall-back: pick standard-time occurrence
    .unwrap_or_else(|| {
        // Gap time (spring-forward): treat naive as UTC
        naive.and_utc()
    });
```

Note: The `.unwrap_or_else` replaces the `?` operator for the gap case. For gap times, `mapped.single()` returns None and `mapped.latest()` also returns None (because `MappedLocalTime::None` yields None from both). The fallback interprets the naive datetime as UTC, matching the existing floating-time fallback pattern in the code.

Remove the `?` from the end of the old line. The function should `return Some(dt.to_utc());` -- but since `naive.and_utc()` is already UTC, and `mapped.single()/latest()` returns a `DateTime<Tz>` that needs `.to_utc()`, the cleanest approach is:

```rust
let mapped = tz.from_local_datetime(&naive);
let dt_utc = mapped.single().map(|dt| dt.to_utc())
    .or_else(|| mapped.latest().map(|dt| dt.to_utc()))
    .unwrap_or_else(|| naive.and_utc());  // gap: treat as UTC
return Some(dt_utc);
```
